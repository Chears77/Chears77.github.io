# 我花了三天，从零搞出了一个记账小程序

> 一个非程序员出身的人，靠 AI 帮忙，把脑子里的想法变成了微信里能跑的小程序。这不是什么技术博客，就是一篇真实记录，告诉你我是怎么做到的，踩了哪些坑，又学到了什么。

---

## 为什么又要做一个记账工具？

这个问题我自己也问过。市面上的记账 App 多到数不清，随手记、挖财、鲨鱼记账……功能一个比一个花哨。但说实话，我用过不下十款，没有一个坚持超过两周。

原因很简单：**太麻烦了**。

每记一笔账要选日期、选分类、填金额、填备注，操作四五步。对我这种懒人来说，这个门槛太高了。我想要的记账方式就一种：像发微信一样，把一天的花销噼里啪啦打出来，系统自动帮我把钱算清楚。

比如我打：

```
6月
1#早餐20
2号游乐场玩水20，跳楼机20，碰碰车30
20260703洗车30元
```

系统应该自动识别出 6 月份的 4 笔支出，分别归到餐饮和娱乐，总额 100 元。

市面上没有这样的工具。那我就自己做一个。

---

## 技术选型：为什么选微信小程序？

说实话，这不是我做的第一个版本。最开始我用 HTML + localStorage 做了一个本地网页版，自己用着还行。但问题来了：手机上打开网页别扭，数据存浏览器里也不放心。

后来我试着接豆包 AI 做智能解析，效果不错但免费额度几天就用完了。

最终决定：**微信小程序**。

理由很简单：不用下载 App，微信里点开就用；数据存在手机本地，不用担心隐私泄露；而且我日常最常用的场景就是微信——聊着天突然想到今天花了什么钱，切过去记一笔就行。

技术栈就是最朴素的组合：**微信小程序原生框架 + Canvas 2D + CSS**。没有用任何第三方 UI 库，没有后端，没有云服务。所有数据都在手机里，用 `wx.setStorageSync` 存，想导出就导出一份 CSV。

---

## 核心功能：智能解析才是灵魂

### 1. 文字输入 → 自动记账

这是整个小程序的核心竞争力。用户只管打字，剩下的交给我写的规则引擎。

输入`1#早餐20`，引擎会自动拆解出：
- 日期：1号（结合上文的月份上下文）
- 项目：早餐
- 金额：20（负数 = 支出，正数 = 收入）

它还能处理更复杂的场景：一行多条记录、带年份的日期、收入支出混记。甚至你写`洗车-30`，它也知道这是支出 30 元。

验证之后我写了一组测试用例：

| 输入 | 解析结果 |
|------|---------|
| `早餐20` | 项目=早餐，金额=-20 |
| `工资+5000` | 项目=工资，金额=+5000 |
| `1#咖啡15，午餐25` | 两笔：咖啡-15 + 午餐-25 |
| `20260703洗车30元` | 日期=7月3日，项目=洗车，金额=-30 |
| `7 月` | 月份上下文，后续无日期记录默认7月 |

### 2. 分类学习：越用越聪明

纯规则引擎有一个死穴：它只能靠关键词猜分类。

比如你记一笔`皮划艇 -100`，引擎看了半天，"皮划艇"是啥？菜名？娱乐项目？猜不出来，只能归到"其他"。

但你手动把它改成"娱乐"，它就记住了。下次再写`皮划艇`，自动归到娱乐。

这个功能我做得比较克制。用户手动调整的时候才记录，自动推理的结果不算数。为的就是避免"越学越错"。

学习记忆有上限（200 条），按关键词长度优先匹配（`皮划艇` 比 `艇` 优先），确保命中的是最准确的规则。

### 3. 对账：给会计思维的人准备的

这是我朋友提的需求。他说："你 App 里账面余额 20，但实际我不仅没钱，还欠信用卡 20。你得把真实资产算进去。"

于是有了对账功能。输入你实际的银行存款、微信余额、现金，系统自动求和，跟账面余额对比，差额就是"对账补遗"——那些你忘了记的、记错了的、或者自己对不上的账。

更贴心的是，你在对账框里写：

```
工行卡 500，微信 230.5，现金-20
```

系统自动提取数字求和，算出实盘 710.5，不需要你心算。

### 4. 统计图表：数字要"看"着舒服

统计页是我改得最多的地方。一开始想用 SVG 画环形图，结果微信小程序的 SVG 支持有坑，`stroke-dashoffset` 传两个值就罢工了。

改成 Canvas 2D 之后顺了很多。现在的统计页长这样：

- **饼状图**：收入为整体，支出分类切片，引线标注分类和占比。如果支出超过收入，超支部分单独一个红色切片。
- **年度柱状图**：12 个月收入支出对比，一眼看出哪个月花超了。
- **分类进度条**：支出按类目排，颜色区分，百分比一目了然。
- **数据洞察**：自动生成分析文字，比如"最大支出是餐饮，占比 45%"，"日均支出 50 元"。

为了这个饼图的引线不交叉，我前后改了五六版。从虚线到贝塞尔曲线，从肘形折线到角度映射。最后用的是按切片角度算 Y 位置，同侧标签做防重叠微调的方案。

## 那些让我崩溃的坑

### 坑一：Babel 转译把 `typeof` 搞崩了

这是最离谱的一个。微信小程序哪怕你关掉 ES6 转译，它还是会用 Babel 处理代码。结果 `typeof x === 'object'` 这种基础语法，被转译成了 `@babel/runtime/helpers/typeof.js` 的调用，而这个模块在小程序环境里根本不存在。

后果就是**整个 App 打开就白屏，所有页面崩溃**。

排查花了我将近一个小时。最终解决方案简单粗暴：把代码里所有 `typeof` 全部替换成等价的判断方式。

### 坑二：Canvas 画饼图，引线打架

这个是我自己设计的问题。饼图边上要放引线标注类别，结果一开始用均匀分布的方式，引线之间互相交叉，指向也看不清楚。

最后拍脑袋想出来的方案：**让每条引线的垂直位置等于它对应切片在圆周上的 Y 坐标**。这样引线从切片自然延伸出来，视觉上不会打架，一眼就能看出哪条线指向哪个切片。

### 坑三：明细页偶尔空白

数据明明写进去了，明细页面就是一片白。排查了好久，最后发现是 WXML 模板里缺了某个闭合标签，导致微信框架根本没注册这个页面。

这些坑说起来都是小事，但堆在一起真的很磨人。每次修好一个，编译，预览，发现问题还在——那种感觉特别无力。好在最终都解决了。

## 说说我为什么坚持本地存储

从头到尾，这个小程序没有任何服务器交互。数据存在微信的本地存储里，不会上传到任何地方。

这个选择是有意的。记账数据太敏感了——你的每一笔花销，本质上是你生活的完整画像。我不想碰用户的数据，也没有能力做好数据安全。

本地存储的代价是：删掉小程序数据就没了。所以我在设置页加了一句话提醒：「请及时保存数据，手机或微信更新后将无法恢复数据」，同时提供了 CSV 导出功能，让用户自己掌握数据。

## 提交审核的那一刻

2026 年 7 月 5 日凌晨，我点了"提交审核"按钮。

说实话有点紧张。之前经历过好几次代码质量扫描没通过——`lazyCodeLoading` 没配、有未使用的页面引用、某些写法不符合规范。一条一条改过来之后，终于看到绿色的"审核中"状态。

能不能过审我不知道，但至少这个小程序是我想要的东西：**打字就能记账，数据留在自己手机上，不用学就会用**。

如果审核过了，欢迎你来试试。如果有想法或者建议，也欢迎告诉我。
