为什么要再做一个记账工具？

市面上的记账 App 多到数不清：随手记、挖财、鲨鱼记账、MoneyWiz……但我用过不下十款，没有一款坚持超过两周。

原因特别简单：太麻烦。

每记一笔账至少要四五步——选日期、选分类、填金额、填备注。对我这种懒人来说，这个门槛直接把我劝退了。

我想要的记账方式就一种：像发微信一样，想到什么花销就噼里啪啦打出来，系统自动帮我把账算清楚。

市面上没有这种工具。那我就自己做一个。

技术选型：为什么是微信小程序

这其实不是我的第一版。最开始我用 HTML + localStorage 做了一个本地网页版，自己能凑合用。但手机浏览器打开网页体验太差，数据存在浏览器缓存里也不踏实。

后来我接了豆包 AI 的 API 做自然语言解析，免费额度几天就耗完了。

最终决定：微信小程序原生框架。

三个理由：

  1. 不用下载 App，微信里打开就用
  2. 数据存本地（wx.setStorageSync），隐私零风险
  3. 日常使用场景就是在微信里——聊着天想到花了什么钱，切过去记一笔

技术栈非常朴素：零第三方库，没有后端，没有云服务。就是原生框架 + Canvas 2D + CSS。所有逻辑都是手写的。

核心功能：智能解析才是灵魂

文字变账单，一行搞定

这是整个小程序的核心竞争力。用户只管打字，我写了一个规则引擎来自动拆解内容。

输入 1#早餐20，引擎自动识出：

  • 日期：1 号（结合上文"6 月"上下文）
  • 项目：早餐
  • 金额：20 元（支出）

你甚至可以一行写多条记录：

你打的字系统解析结果
早餐20项目=早餐，金额=-20
工资+5000项目=工资，金额=+5000
1#咖啡15，午餐25咖啡-15 + 午餐-25
20260703洗车30元日期=7月3日，项目=洗车，金额=-30
7 月设置月份上下文（后续无日期记录默认 7 月）

分类学习：越用越聪明

纯规则引擎有个死穴：它只能靠关键词猜分类。

比如你写 皮划艇 -100，引擎琢磨半天——"皮划艇"是菜名？娱乐项目？猜不出来，先归到"其他"。

但当你手动把它改成"娱乐"，系统就记住了。下次再写 皮划艇，自动归到娱乐。

这个功能我做得比较克制：只在用户手动调整时记录，自动推理的结果不算数。为的就是避免"越学越错"。

学习记忆上限 200 条，匹配时优先用最长的关键词（比如"皮划艇"的优先级高于"艇"），保证命中率。

对账：给你一个"真实"的视角

这是朋友给我提的需求。他的原话是："你 App 显示我账面余额 20 块，但实际上我不仅没钱，还欠信用卡 20 块。你得让我输真实资产。"

于是有了对账功能。在明细页点「对账」，弹出输入框：你写下工行卡余额、微信余额、口袋里现金，系统自动提取数字求和。差额就是"对账补遗"——那些你忘了记的、记错金额的、或者自己对不上的账。

饼图为什么这么难画

一开始我想用 SVG 画环形图。结果微信小程序的 SVG 支持有坑——stroke-dashoffset 传两个值就直接罢工，环形图一片空白。

改成 Canvas 2D 之后好多了。但下一个问题来了：饼图边上的引线互相打架。

五六个分类标签挤在饼图右侧，引线交叉成蜘蛛网。我一版一版地改：

  • 第 1 版：虚线加圆点，丑。
  • 第 2 版：贝塞尔曲线，线太弯了像面条。
  • 第 3 版：肘形折线，看着还行但所有标签挤在右边。
  • 第 4 版：左右两侧分开放，但均匀分布还是交叉。
  • 第 5 版：按切片角度算出 Y 坐标，同侧做防重叠微调。引线从切片自然延伸到标签，视觉上不打架，一眼看出哪条线指向哪个切片。

还搞了一个自适应算法：画布宽度、饼图半径、引线长度都按屏幕尺寸动态计算，确保在小屏手机上文字也不会溢出。

饼图的两个模式

正常情况（收入 ≥ 支出）：饼基 = 收入，支出分类 + 结余切片填满整个圈。

超支情况（支出 > 收入）：饼基 = 支出 + 超支，支出分类 + 红色超支切片填满整个圈，中心显示"189%"，意思是花掉了收入的 1.89 倍。

那些让我崩溃的坑

坑一：Babel 转译，把 typeof 搞崩了

🐛 Bug 级别：全页面白屏

微信小程序哪怕关掉 ES6 转译，它的工具链还是会用 Babel 处理代码。结果 typeof x === 'object' 被转译成 @babel/runtime/helpers/typeof.js 的调用，这个模块在小程序环境里压根不存在。

后果：整个 App 打开就白屏，所有页面同时崩溃。

排查花了一个小时。最终方案：把代码里所有 typeof 全部替换成 .constructor 等价的判断方式。

坑二：明细页偶尔空白

🐛 Bug 级别：数据在但页面不渲染

数据明明写进了 storage，明细页就是一片白。排查半天发现是 WXML 模板里缺了某个闭合标签，微信框架根本没注册这个页面。

教训：标签不闭合，整个页面直接消失了——微信小程序的容错比想象中差很多。

坑三：金额字段在手机上无法编辑

明细页的金额一开始用 <view> 展示，后来想改成可编辑的 <input>。结果 type="digit" 在部分手机上唤起数字键盘后再也收不回去。最后改成 type="text" 配合 bindblur 事件，才兼容了所有设备。

坑四：小程序工具链的隐蔽行为

微信开发者工具在"代码质量"扫描时会检查你写了哪些页面但没有在 app.json 中注册，还会检查组件是否按需加载。忘记配 lazyCodeLoading: "requiredComponents" 就直接扣分。

这些都不是功能 bug，但审核过不了也很头疼。

✦ ✦ ✦

为什么我坚持本地存储

从头到尾，这个小程序没有任何服务器交互。记账数据存在微信的本地存储里，不会上传到任何地方。

这个选择是有意的。记账数据太敏感了——你的每一笔花销，本质上是你生活的完整画像。我不想碰用户的数据，也没有能力做好数据安全。

本地存储的代价是：删掉小程序数据就没了。所以我在设置页加了一句话：

「请及时保存数据，手机或微信更新后将无法恢复数据。」

同时提供了 CSV 导出功能，让用户自己掌控数据。

提交审核的那一刻

2026 年 7 月 5 日凌晨，我点了"提交审核"按钮。

说实话有点紧张。之前经历过好几次代码质量扫描没通过。一条一条改过来之后，终于看到审核状态变成"审核中"。

能不能过审我不知道。但至少这个小程序是我想要的东西：

打字就能记账，数据留在自己手机上，不用学就会用。

如果审核过了，欢迎你来试试。如果有想法或者建议，也欢迎告诉我。

如果你也想做一个

我的体会是：工具在变，但做产品的核心逻辑没变。

先想清楚你到底在解决谁的什么问题。我做的不是"又一个记账 App"，而是一个"打字就能记账"的工具。这个差异化让我有动力把它做完。

然后别怕技术门槛。说实话我写第一行代码的时候，连微信小程序的框架长什么样都不知道。遇到问题就查文档、看社区、用 AI 帮忙。现在的 AI 工具真的很强，能帮你把脑子里的想法快速变成可运行的代码。

最后——先做出来，再优化。我的统计页改了六版才满意，第一版其实也能用。与其纠结一个完美的方案迟迟不动手，不如先把能跑的版本搞出来。
