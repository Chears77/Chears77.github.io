# 第二版（3第二版）构建与部署说明

本说明随 3第二版 一并存放，使第二版完全自包含、与第一版（2第一版）脱钩。
第二版是**网站实时数据源 + 站点产物**的唯一来源，改库/重建/部署一律以 3第二版 为准。

## 法规动态说明覆盖情况（已完成）

- 修订类 **114 个**：联网核对国务院/人大公报，写入完整「历次修订链」（第 N 次修订）。
- 非修订 NEED 类 **155 个**：自动生成「公布/发布日期 + 机关 + 文号，自施行日期起施行」。
- 中央规范性文件 **49 个**：自动生成「印发日期 + 机关 + 文号，自施行日期起施行」。
- 司法案例 / 行政案例 / 政策解读（doc_type:doc）正确跳过。
- 校验：坏 FM = 0、缺 H1 = 0、NEED 缺说明 = 0。

## 本地构建（改完 法规库/ 后必须重跑）

```bash
cd 3第二版
python build_data.py        # 重建 laws/ + data/manifest.json + data/search.json
# 若改了前端逻辑/样式（gen_website.py 源），再跑：
python build_appjs.py
python build_css.py
python gen_nav.py
```

## 本地预览（必须经 HTTP，file:// 不行）

```bash
cd 3第二版
python serve.py            # 默认 8123，浏览器打开 http://localhost:8123/index.html
# 或直接：python -m http.server 8123
```

> 双击 index.html（file://）会被浏览器安全策略拦截 fetch，报 "Failed to fetch manifest.json"，属正常，非 bug。

## 部署到 GitHub Pages（关键：Jekyll 问题）

1. 把 **3第二版 整个目录内容**复制到 GitHub 仓库对应目录（本仓库部署在 `gongchengfagui/` 子目录）。
   必须包含：`laws/`、`data/`、`assets/`、`index.html`、`nav.html`、`.nojekyll`。
   `法规库/` 中文源线上不读取，可不传省体积。
2. **仓库根目录必须存在 `.nojekyll`**（已加）。原因：GitHub Pages 默认启用 Jekyll，
   Jekyll 会处理 560 个带 front matter 的 `laws/*.md` 导致构建失败，页面停留在旧构建
   （表现为法规数量 503、打开法规 404）。`.nojekyll` 禁用 Jekyll 后，Pages 直接静态托管
   所有文件，`laws/*.md` 才能被前端正常 fetch。
3. 提交并推送：`git add -A && git commit -m "..." && git push`。
4. 推送后到仓库 **Settings → Pages** 确认构建成功（无 "build failed"），等待 1~2 分钟。
5. 浏览器访问时用 **Ctrl+Shift+R 硬刷新** 清缓存。

## 与 法规库/ 中文路径的 404 区别

早期版本 manifest 的 `file` 指向 `法规库/中文名.md`，GitHub Pages 对中文路径
percent-encode 后 fetch 返回 404（本地 Python 服务器反而 200，故只在线上报）。
现 manifest 已改为指向 ASCII 的 `laws/...md`，彻底解决。线上再报 404 必查：
① `laws/` 是否随本次构建部署；② 仓库根 `.nojekyll` 是否存在。
